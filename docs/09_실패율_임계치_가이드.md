# 실패율 임계치 가이드

> 대상: 로그인/투표/알림/회비확정/예약보고 핵심 액션

---

## 1. 핵심 지표 정의

- `요청 수`: 특정 액션 호출 총 건수
- `실패 수`: 해당 액션 실패 건수
- `실패율`: `실패 수 / 요청 수`
- `오류 코드 분포`: `permission-denied`, `invalid-argument`, `failed-precondition` 등

---

## 2. 권장 임계치 (초기)

| 액션 | 경고(Warning) | 심각(Critical) | 비고 |
|------|---------------|----------------|------|
| 로그인 | 3% 초과(10분) | 8% 초과(5분) | 인증/플랫폼 이슈 |
| 투표/출석 저장 | 2% 초과(10분) | 5% 초과(5분) | UX 영향 큼 |
| 회비 확정 | 1% 초과(10분) | 3% 초과(5분) | 민감 액션 |
| 예약 결과 보고 | 2% 초과(10분) | 5% 초과(5분) | 운영 영향 큼 |
| 미납 알림 발송 | 3% 초과(10분) | 10% 초과(5분) | 토큰/권한 이슈 |

---

## 3. 코드별 해석 가이드

- `unauthenticated`
  - 로그인 세션 만료/클라 인증 누락
- `permission-denied`
  - 역할 매핑 오류, Rules/함수 권한 미스매치
- `failed-precondition`
  - App Check 미검증, 상태 전이 불가(중복 성공 처리 등)
- `invalid-argument`
  - 클라이언트 payload 버전 불일치, requestId 누락
- `resource-exhausted`
  - rate limit 초과 (과호출/오사용 패턴)
- `internal`
  - 서버 예외/데이터 정합성 문제 가능성

---

## 4. 대응 플레이북 (요약)

### Warning 진입
- 10분 모니터링 강화
- 오류 코드 상위 1~2개 원인 확인
- 기능 토글 유지, 사용자 영향 메시지 준비

### Critical 진입
- 즉시 완화: 관련 기능 토글 OFF
- 최근 배포/설정 변경 점검
- 15분 내 임시 정상화 목표

---

## 5. 데이터 소스

- 서버 실패 집계: `teams/{teamId}/ops_metrics_daily/{yyyy-mm-dd}`
  - `failures.{action}.total`
  - `failures.{action}.codes.{errorCode}`
- 감사 로그: `teams/{teamId}/audit_logs`

---

## 6. 초기 운영 팁

- 출시 1주차는 임계치를 다소 완화해 오탐을 줄인다.
- 2주차부터 실제 트래픽 기반으로 임계치 재설정한다.
- 임계치 조정 이력도 `audit_logs` 또는 운영 문서에 남긴다.
